------------------------------------------------------------------------
  FILE: backend/models.py
------------------------------------------------------------------------
PURPOSE: Defines all data shapes (Pydantic models) that the API uses.

WHAT WAS ADDED:
  - RegisterRequest   : name, email, password, role — for /register endpoint
  - RegisterResponse  : success, message, role
  - LoginResponse     : added `mfa_required: bool` field (new)
      When True, frontend must show OTP entry page before granting access.
  - SetupMFAResponse  : returns totp_uri (for QR code) and secret (backup key)
  - ConfirmMFARequest : email + code — user submits first OTP to activate MFA
  - VerifyMFARequest  : email + code — OTP submitted during login step 2
  - MFAResponse       : success, message, role — generic MFA result

------------------------------------------------------------------------
  FILE: backend/database.py
------------------------------------------------------------------------
PURPOSE: All database operations. Completely rewritten for security.

WHAT CHANGED:

  PASSWORDS — bcrypt
    - hash_password(plain)      : hashes a password with bcrypt before storing
    - verify_password(plain, stored) : backward-compatible verification:
        * If stored hash starts with "$2b$" → use bcrypt.checkpw()
        * Otherwise → plain text comparison (for any existing un-hashed users)
      WHY: This migration strategy means old accounts keep working during
           the transition period without any manual DB update needed.

  REGISTRATION — register_user()
    - Checks for duplicate email first (raises ValueError if found)
    - Hashes the password with bcrypt before inserting
    - Pre-generates a pyotp TOTP secret (stored in mfa_secret column)
      MFA is NOT active yet — user must scan QR code and confirm first.

  MFA FUNCTIONS
    - get_mfa_secret(email)  : fetches stored TOTP secret for a user
    - enable_mfa(email)      : sets mfa_enabled = TRUE in DB
    - verify_totp(email, code) : verifies the 6-digit code using pyotp.TOTP
      valid_window=1 means ±30 seconds tolerance (handles clock skew)

  AUDIT LOGGING
    - log_login(email, role) : inserts a row into Login_Log on every
      successful FULL login (only after MFA is also verified if enabled)

------------------------------------------------------------------------
  FILE: backend/main.py
------------------------------------------------------------------------
PURPOSE: FastAPI server — all HTTP endpoints. Significantly updated.

RATE LIMITING (slowapi)
    slowapi wraps FastAPI and counts requests per IP address.
    @limiter.limit("5/minute") on /login and /verify-mfa endpoints.
    If exceeded → HTTP 429 Too Many Requests automatically.
    WHY: Prevents brute-force password and OTP guessing attacks.

NEW / CHANGED ENDPOINTS:

  POST /login (CHANGED)
    Step 1 of the two-step login process.
    - Verifies email exists in DB
    - Uses verify_password() (bcrypt-aware)
    - If mfa_enabled=True: returns {mfa_required: True} — does NOT log in yet
    - If mfa_enabled=False: logs the login via log_login() and returns role
    - Rate limited: 5 attempts per IP per minute

  POST /register (NEW)
    - Validates role is 'institution' or 'company'
    - Calls register_user() which bcrypt-hashes the password
    - Returns 409 Conflict if email already registered

  GET /setup-mfa/{email} (NEW)
    - Fetches user's pre-generated mfa_secret from DB
    - Builds the TOTP URI using pyotp:
        otpauth://totp/CertVerify:email?secret=XXX&issuer=CertVerify
    - Frontend renders this URI as a QR code (using react-qr-code)
    - User scans it with Google Authenticator or Authy app

  POST /confirm-mfa (NEW)
    - Called once after user scans QR and enters their first code
    - Verifies the 6-digit code
    - Calls enable_mfa(email) to set mfa_enabled=True in DB
    - From this point on, every login requires the OTP

  POST /verify-mfa (NEW)
    Step 2 of the two-step login process (only reached if mfa_required=True).
    - Verifies the 6-digit TOTP code against stored secret
    - If correct: calls log_login() and returns role
    - Rate limited: 5 attempts per IP per minute

  GET /login-history (NEW)
    - Returns last 20 rows from Login_Log table
    - Shows: email, role, logged_in_at
    - Intended for the admin Dashboard

  AUTH PAGES
------------------------------------------------------------------------

  src/pages/signin.tsx (REWRITTEN — was a simple sign-in form)
    NOW  : Portal Selector landing page.
    SHOWS: Two cards — "Institution Portal" and "Company Portal"
    Each card has Sign In and Register buttons linking to role-specific pages.
    WHY  : Separating portals makes it impossible to accidentally log in

  src/pages/InstitutionSignIn.tsx (NEW — replaced generic signin logic)
    WHAT : Sign-in form specifically for institution accounts.
    FLOW :
      1. User submits email + password
      2. POST /login called
      3. If role ≠ 'institution' → error shown ("use Company portal")
      4. If mfa_required=True → navigate to /verify-mfa with email in state
      5. If success → AuthContext.login('institution') + navigate /institutions
    SHOWS: "MFA protected" in branding panel
           Success banner if arriving after MFA setup (router state flag)

  src/pages/InstitutionSignUp.tsx (NEW)
    WHAT : Registration form for institution accounts.
    FLOW :
      1. User fills name, email, password, confirm password
      2. POST /register with role='institution'
      3. On success → navigate to /setup-mfa (MFA setup is MANDATORY)
    SHOWS: Warning notice: "MFA required after registration"

  src/pages/CompanySignIn.tsx (NEW)
    WHAT : Sign-in form for company accounts (blue/indigo branding).
    FLOW :
      1. POST /login
      2. If role ≠ 'company' → error
      3. On success → navigate to /verify (companies only verify docs)

  src/pages/CompanySignUp.tsx (NEW)
    WHAT : Registration form for company accounts.
    FLOW :
      1. POST /register with role='company'
      2. On success → navigate to /signin/company
    NOTE: Companies do NOT go through MFA setup (optional, not forced).

  src/pages/MFASetupPage.tsx (NEW)
    WHAT : One-time MFA setup page shown after institution registration.
    HOW  :
      1. GET /setup-mfa/{email} → receives totp_uri and secret
      2. Renders QR code using <QRCode value={totp_uri} /> from react-qr-code
         (The QR code encodes: otpauth://totp/CertVerify:email?secret=...)
      3. Shows the raw secret text with a Copy button (for manual entry)
      4. User enters first 6-digit code to confirm → POST /confirm-mfa
      5. On success → navigate back to /signin/institution with success flag
    LIBRARY: react-qr-code (lightweight, SVG-based, no network calls)

  src/pages/MFAVerifyPage.tsx (NEW)
    WHAT : Step 2 of login — enter OTP after password is verified.
    HOW  :
      - Receives email + role via React Router location.state
      - User opens Google Authenticator, enters 6-digit code
      - POST /verify-mfa → on success → AuthContext.login(role)
      - Redirects institutions to /institutions, companies to /verify
    UX  : Large monospace input field, numeric keyboard on mobile

  → /signup/company: fill form → POST /register (role='company')
  → navigate /signin/company

  /signin/company: email + password → POST /login
  → response: {success: true, mfa_required: false}

  Any user → navigate to /dashboard or /institutions
  → ProtectedRoute checks: not logged in? → redirect /signin
  → ProtectedRoute checks: wrong role? → "Access Denied" screen


================================================================================
  SECTION 6: SECURITY LAYERS SUMMARY
================================================================================

  Layer 1  bcrypt password hashing
           All new passwords are hashed before storing. Even if the DB is
           leaked, passwords cannot be recovered without brute force.

  Layer 2  TOTP MFA (mandatory for institutions)
           Institutions must use Google Authenticator. A stolen password
           alone cannot grant access.

  Layer 3  Rate limiting
           /login and /verify-mfa: max 5 requests/min per IP address.
           Prevents automated brute-force attacks.

  Layer 4  Role-based access control
           Companies cannot see the database even if they have valid credentials.
           Enforced at both route level (ProtectedRoute) and nav level (Navbar).

  Layer 5  Login audit logging
           Every successful login writes email, role, and timestamp to Login_Log.
           Suspicious access patterns become visible.

  KNOWN LIMITATIONS (for future improvement):
    - JWT tokens would be more secure than storing role in localStorage
    - Session expiry / auto-logout is not yet implemented
    - Email verification after signup is not yet implemented
    - MFA setup is optional for companies (can be made mandatory)
